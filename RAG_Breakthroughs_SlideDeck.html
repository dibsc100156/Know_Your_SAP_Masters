#2 Hybrid Search

Sparse (BM25) + Dense (Vector) via Reciprocal Rank Fusion
🔑 BM25 / SPLADE — exact keyword match 🧠 Dense vectors — semantic understanding ⚖️ RRF fusion — rank-based weighting

The Innovation

  • Sparse retrieval (BM25) — TF-IDF based, finds exact terms like "iPhone 15", "C++ error 0x8004"
  • Dense retrieval (OpenAI/BGE embeddings) — understands synonyms, intent, cross-lingual
  • RRF (Reciprocal Rank Fusion) — merges rankings without score normalization issues

Real-World Impact

  • Vanguard (global asset manager): +12% retrieval accuracy by tuning alpha weight
  • Financial docs have exact codes, abbreviations — pure vector search misses them
  • Production-ready: Pinecone, Weaviate, Elasticsearch all support hybrid today

#3 Reranking — Cross-Encoder Last Mile

Precision gating after initial broad retrieval
⚡ Bi-Encoder → Query/Doc independently 🔥 Cross-Encoder → Query+Doc together 📦 Funnel Mode: Top-100 → Top-5

The Innovation

  • Initial retrieval is broad but imprecise — speed over accuracy
  • Cross-Encoder concatenates Query + Document as input — captures subtle interactions
  • Scores re-ranked, Top-5 fed to LLM — much higher precision

Performance Gains

  • Databricks Mosaic AI: Recall@10: 74% → 89%
  • Added latency: ~1.5 seconds for 50 docs (acceptable for async)
  • Cohere Rerank v3 and BGE-Reranker are top choices
  • FlashRank for ultra-low latency use cases

#4 HyDE — Hypothetical Document Embeddings

Generate hypothetical answer → embed that → retrieve real docs
💡 LLM generates "ideal answer" first 📐 Embed the hypothetical — not the query 🎯 Closer semantic distance to real document

The Innovation

  • User query → LLM generates a plausible, well-structured hypothetical answer
  • This hypothetical answer is embedded (not the short user query)
  • The hypothetical document is semantically closer to the real target document in vector space
  • Works especially well for technical, structured Q&A

Why It Works

  • Short queries ("Q3 financial anomalies") are semantically distant from long documents
  • A generated answer has similar length and structure to real documents
  • Particularly powerful for Stack Overflow / technical forums
  • Can be combined with any embedding model

#5 Corrective RAG (CRAG)

Self-correcting retrieval via confidence-gated quality check
🔍 Post-retrieval relevance evaluation 🌐 Low confidence → web search fallback 🗑️ Discard wrong docs before generation

The Innovation

  • After retrieval, a lightweight LLM evaluator scores the relevance of each chunk
  • If confidence is below threshold → trigger correction workflow
  • Correction can be: re-phrase query, expand to web search, or use alternative index
  • Acts as a quality gate — prevents garbage-in → garbage-out

Why It Matters

  • Naive RAG blindly trusts retrieval — even wrong chunks go to LLM
  • CRAG distinguishes "good enough" from "hallucination-prone"
  • Can be combined with Self-RAG (token-level critique) for double safety
  • Production essential for high-stakes domains (legal, medical, finance)

#6 Self-RAG — Reflection Tokens

End-to-end self-verification using reflective critique tokens
🪞 Critique tokens — IsRET, IsREF, IsSUP 🤖 Model judges its own retrieval quality 🚫 Suppress irrelevant generations

The Innovation

  • Trained models (or prompted) emit special reflection tokens during generation
  • IsRET: Is the retrieved context relevant?
  • IsREF: Does the response require retrieval?
  • IsSUP: Is the generation fully supported by the context?
  • Tokens are parsed at inference — irrelevant generations are suppressed

Why It Beats CRAG

  • CRAG needs an external evaluator LLM call
  • Self-RAG bakes the critique into the model's forward pass
  • No additional LLM call overhead during inference
  • Achieves lower hallucination rates than externally gated systems

#7 Agentic RAG

Planning, Tool Use & Autonomous Orchestration — RAG as an Agent
🧠 Planning — break complex queries into steps 🔧 Tool use — SQL, API, web, graph traversal 🔄 Multi-step reasoning with memory

  • Given a complex question, it plans sub-steps, calls tools, evaluates intermediate results

#8 GraphRAG — Knowledge Graph + Hierarchical Communities

Microsoft Research | Leiden community detection + Map-Reduce summarization
🕸️ Knowledge graph over corpus 🏘️ Leiden community detection 🗺️ Map-Reduce for global questions

The Innovation

  • LLM extracts Entities + Relationships from all documents → builds a knowledge graph
  • Leiden algorithm detects tightly connected communities (better than Louvain)
  • Each community gets a pre-computed natural language summary
  • Map-Reduce: For global questions → fetch community summaries → map intermediate answers → reduce to final

Solves the "Global Question" Problem

  • Naive RAG cannot answer: "What are all the conflicts in Q2?"
  • GraphRAG pre-computes community summaries — answers span entire corpus
  • LinkedIn case: drastically improved relevance on HR policy questions
  • ⚠️ $33K indexing cost for large datasets — Local GraphRAG is the open-source fix

#9 Vectorless RAG

Deterministic Retrieval — SQL, Graph & Keyword Search without Embeddings
⚡ SQL-first retrieval — live DB queries 🕸️ Graph traversal — deterministic hops 🔑 BM25 / Elasticsearch — exact keyword

The Innovation

  • Text-to-SQL RAG: LLM writes SQL → executes against live DB → exact rows as context
  • Graph RAG: Traverse relationships (e.g., MARA → LFA1) — deterministic, no hallucination
  • Keyword RAG: BM25 / Elasticsearch for exact SKU, part number, tax code matching
  • Completely bypasses dense embeddings — no embedding cost, no vector search latency

Why It's Critical for Enterprise

  • Vector search is unreliable for exact IDs: "MAT-9921" vs "MAT-9922"
  • Can do MATH (SUM, AVG, COUNT) — vector stores cannot
  • Fully explainable: "I queried table LFA1 where LIFNR = '001'"

#10 Hierarchical RAG

Tree-Structured Retrieval — Module → Transaction → Field (SAP-native)
🌲 Multi-level tree indexing 👆 Parent pointers preserve structure 🎯 Top-down progressive narrowing

The Innovation

  • Documents chunked into a tree: L0 (root/summary) → L1 (section) → L2 (subsection) → L3 (leaf chunks)
  • Each level has optimized index: L0=BM25, L1=small vectors, L3=large vectors
  • Retrieval narrows progressively: "Which module?" → "Which transaction?" → "Which field?"
  • Parent pointers restore structural context — no more "lost in the middle"

SAP-Native Fit

  • SAP itself is hierarchical: Module (MM) → Transaction (ME21N) → Screen/Field (EKPO)
  • Only relevant subtree is loaded — no noisy global search
  • Answers cite: [Module 3.2.1, Chunk 4] — fully traceable
  • Perfect match for our "Know Your SAP Masters" 5-Pillar architecture!
